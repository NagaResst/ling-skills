# holding-adjustment：组合仓位调整统一入口

## 三层架构

```
┌─────────────────────────────────────────────────┐
│ Layer 0：入口路由（判断用户在哪一层）              │
├─────────────────────────────────────────────────┤
│ Layer 1：组合前置诊断（所有动作必经）              │
│   • 画像校准 + 持仓快照 + 仓位变化来源表           │
├─────────────────────────────────────────────────┤
│ Layer 2：整体再平衡                              │
│   • 4 档框架 → 时代主题修正 → 拆解标的 → 画像补丁  │
├─────────────────────────────────────────────────┤
│ Layer 3：单只基金操作                            │
│   • 止盈止损线 + 含费成本 + 置换 + 去重补缺        │
└─────────────────────────────────────────────────┘
```

**进入方式**：用户问"某只基金怎么办"或"调仓"时，无论最终落到哪一层，**都必须先走 Layer 1**（组合前置诊断）→ 再决定展开 Layer 2 或 Layer 3。

**核心原则**：先确认后计算，每步对账，每层闭环。**用户是决策者，AI 是计算器和数据收集员。**

## Layer 0：入口路由

用户问的可能是以下三种之一：

| 用户典型问题 | 走的层 | 触发关键词 |
|------------|-------|-----------|
| "我该不该补光模块" / "现在比例怎么调" | Layer 1 → Layer 2 | 调仓、再平衡、比例、补仓、4档、画像 |
| "这只基金要不要卖" / "止盈线在哪" | Layer 1 → Layer 3 | 单只、止盈、止损、赎回、退出、置换、换基 |
| "组合里重复了哪些" / "我缺什么方向" | Layer 1 → Layer 3 | 去重、补缺、暴露、产业链 |

**不确定时**：先问用户"是想做整体调仓，还是只看某一只基金"。

## Layer 1：组合前置诊断（所有动作必经）

### 1.1 必读输入

1. `投资者行动/投资者画像.md` — 目标结构和约束
2. `投资者行动/持仓情况.md` — 真实持仓份额
3. `投资新闻归档/YYYY-MM/YYYY-MM-DD/raw_data/market_momentum_YYYY-MM-DD.json` — 当日市场数据
4. `投资新闻归档/YYYY-MM/YYYY-MM-DD/raw_data/analysis_snapshot_YYYY-MM-DD.json` — 持仓快照

### 1.2 组合校准三件事

- **总资产 = 基金市值 + 现金**（余额宝+银行卡+其他货币基金）。报告里的"组合总持仓 X 元"= 基金市值合计，不含现金，不能直接当总资产用。**占比分母 = 现金+基金市值**。
- **持仓情况.md 与最新报告可能不同步**。以最新报告的份额/成本为准，告诉用户"持仓情况.md 上次更新是 X，最新报告是 Y，要不要回写？"。
- **未建仓/份额未填的标的**：份数栏空白按 0 计入，明示"X 标的份数待补，未计入"。

### 1.3 画像四大类（默认中枢）

| 档位 | 区间 | 中枢 |
|------|------|------|
| 现金+货币 | 15-25% | 20% |
| 稳健防御 | 28-35% | 30% |
| 核心 Beta | 20-30% | 25% |
| 政策主线进攻 | 20-30% | 25% |

**硬约束：4 档合计必须 = 100%**。任何"画像外"持仓（如家电、消费等）必须挤进 4 档的某一档，不能独立存在。

### 1.4 仓位变化来源表（强制输出）

凡正文中出现"仓位变重 / 变轻 / 权重抬升"等表述，必须同步给出一份可复核的来源表，至少覆盖被点名的基金或结构层：

| 基金名称(代码) | 前一快照金额 | 当前金额 | 金额变化 | 份额变化方向 | 净值贡献 | 份额贡献 | 主导因素 |
|---------------|------------|---------|---------|------------|---------|---------|---------|

硬规则：
1. 若份额未变，不得把金额或权重上升写成主动加仓。
2. 若份额增加但金额反而下降，写成"真实加仓，但被净值下跌抵消"。
3. 若份额减少但金额下降更快，写成"真实减仓叠加净值下跌"。
4. 新建仓必须单列，不得与普通加仓混写。

### 1.5 验证用户前提（关键步骤）

**当用户连续给出 ≥ 3 条约束规则**（如"红利 0% / 债基换低波 / 绿电算进攻 / 家电持有到 9 月"），**不要直接套规则开算**，必须先：

1. 用列表回显用户说过的全部规则，一条不漏
2. 问 "我认可 / 验证 / 还有要补的吗"
3. 区分**硬约束**（不可违反）和**软偏好**（可微调）
4. **得到确认后再开算**

## Layer 2：整体再平衡

### 2.1 工作流

```
阶段零：读画像与现状（Layer 1 输出）
    ↓
阶段零·五：验证用户前提
    ↓
阶段一：用户拍 4 档目标比例
    ↓
阶段一·五：板块基 vs 主动基 分类
    ↓
阶段二：用户拍档内拆解
    ↓
阶段二·五：限购 / 申赎 限制检查
    ↓
阶段三：用户拍每个调仓动作
    ↓
阶段四：对账与归档
```

### 2.2 阶段一：用户拍 4 档目标

**不要替用户选 4 档比例**。问清楚：
- 现金档具体放多少（15 / 18 / 20 / 25%）
- 防御档放多少（28% 下限 / 30% 中枢 / 35% 上限）
- 核心 Beta 放多少（20% 下限 / 25% 中枢 / 30% 上限）
- 进攻 = 100% - 上面三档

如果用户说"按画像走"，默认中枢：现金 20% / 防御 30% / 核心 Beta 25% / 进攻 25%。但**任何"主题修正"（上升期/震荡期/防御期）都让用户拍**。

### 2.3 阶段一·五：板块基 vs 主动基 分类

| 类型 | 当前 9 只里的例子 |
|------|------------------|
| **板块基**（被动追踪指数） | 汇添富中证芯片产业指数增强 C、富国中证绿色电力 ETF 联接 A、易方达中证家电龙头 ETF 联接 C、天弘中证科创创业 50 ETF 联接 A、中银 MSCI 中国 A50 互联互通指数增强 A |
| **主动基**（经理选股） | 华商新趋势、财通集成电路、财通新视野、国联安科技动力、财通成长优选 |

让用户拍"进攻档要全部主动基 / 允许板块基 / 给定比例"。进攻档内**优先用主动基承担**，板块基仅在用户明确要求时进入。

### 2.4 阶段二：用户拍档内拆解

**不要替用户决定每只基金的金额**。问清楚：
- 核心 Beta 3 只宽基各放多少（等额 / 按当前比例 / 按夏普加权 / 集中 1-2 只）
- 进攻档内子板块各放多少（芯片 / 光模块 / 整机 / PCB / 绿电 / 家电 / 弹性）

**子板块排序规则**（用户视角，不是基金报告标签）：
- "半导体" = 芯片 + 设备 + 材料 + 封测 + IDM
- "光模块" = 光通信 + CPO + 光器件 + 光芯片 + 光纤
- "PCB" = 覆铜板 + 封装 + 基材 + 精密加工
- "整机服务器" = 服务器 + 数据中心 + 算力基础设施
- "网络/IDC" = 通信设备 + IDC + 液冷
- "绿电" = 绿电运营商 + 数据电力
- "家电" = 家电龙头 + 家电出海

让用户拍"哪个段应该占比最高"，不要替用户排序。

候选方案**只能来自用户报告**或用户已持有的基金。缺数据时明示"报告缺失, 需 akshare 拉取"或"用指数级数据替代"，让用户拍。

### 2.5 阶段二·五：限购 / 申赎 限制检查

推荐具体新进基金时：
1. 用 `terminal` 跑 `ak.fund_open_fund_info_em` 查基金概况（sandbox python 没有 akshare，走 terminal 系统 python）
2. 检查字段"单笔单日限额" / "单日限额" / "限购"
3. 如果有限购 → 明示"需 N 天分批建仓 / 期间追高或接刀风险"，给 1-2 个收尾选项

### 2.6 阶段三：用户拍每个调仓动作

给出每只基金的：
- 当前市值 → 目标市值
- 补仓金额
- 新 % 占比
- 新份数（按最新净值）

等用户确认后，才写"对账"段落。

**收尾纪律**：
- 如果用户选的方案**超过画像上限/下限** → 主动附 1-2 个收尾选项
- 如果调仓后**净现金释放 > 缺口** → 明示剩余未分配金额

### 2.7 阶段四：对账与归档

4 档差额 = 释放来源 - 加仓去向，全部对账，差 0.1% 以下可忽略。

### 2.8 调仓行为类型（先判断属于哪种类型）

| 类型 | 触发 | 例子 | 分析侧重点 |
|------|------|------|-----------|
| **止盈再观察型** | 仓位偏重 / 短期有利润 / 政策逻辑没变 | 财通集成电路从 8% 减至 4% 之前的过渡阶段 | 减仓后**回补规则** |
| **工具替换型** | 经理风格 / 产品形态（板块基 vs 主动基）与目标不匹配 | 汇添富中证芯片产业指数增强 C → 财通新视野 A | **新工具的暴露结构** vs 旧工具 |
| **风格不兼容型** | 工具/主题（红利）与主线风格不兼容 | 天弘中证央企红利 50 指数 A vs AI 算力进攻主线 | **完全退出** |

### 2.9 画像补丁（Layer 4：调仓后规则升级）

**目标**：把这次调仓暴露的画像规则缺口补进 `投资者画像.md`。

触发条件：
- 调仓中触发了画像没说清楚的规则
- 用户在对话中**主动纠正**了画像某条规则
- 用户否决了画像某条默认行为

步骤：
1. 列出这次调仓暴露的画像缺口（每条引用对话原文）
2. 提议补丁（位置 + 内容 + 优先级）
3. 等用户拍哪些补丁通过
4. 用户否决的补丁：标注"已讨论但未采纳"
5. 写入 `投资者行动/投资者画像.md`：
   - 第二部分「六、给 AI 的默认读取口径」插入补丁条目
   - 文件底部版本号 +1（如 v3.0 → v3.1）

补丁格式：
```markdown
- **【v3.x 补丁 N】<规则名>**：<内容>
```

### 2.10 数据纪律

- **拉取基金历史数据时**：先问用户要什么区间。如果用户没明确，默认最近 1 年。**绝不能默认拉"成立至今"**——会混入熊市最深那段。
- **推荐具体基金时**：只能用用户 `基金研究报告/` 目录里已有的报告。
- **报告缺失时**：明示"你报告里没有专门的XX基金, 选项 A/B/C 来自你报告里重仓XX的现有基金"，让用户拍板。

## Layer 3：单只基金操作

### 3.1 必读输入

Layer 1 + 目标基金的 `基金研究报告/<基金全称>/`（如有）。

### 3.2 核心判断框架（五类必答信号）

#### 信号 1：组合位置是否仍合适

重点看：
- 画像里建议组合的目标结构是什么
- 当前组合总诊断是防守偏厚、宽基偏薄、还是主题重叠偏高
- 这只基金承担的是稳定器、核心宽基、主题仓还是观察仓
- 当前仓位是否符合组合建议

#### 信号 2：原逻辑是否还在

- 原正证是否还成立
- 原风险是否扩大
- 是否出现新的反证

#### 信号 3：当前强弱是否支持继续拿

- 近 1 月超额收益
- 回撤后修复力度
- 是否持续弱于主线 ETF

#### 信号 4：确认级别是否足够

- 是否只有单日修复
- 是否缺少连续确认
- 是否存在关键数据缺口

#### 信号 5：信息边界在哪里

常见边界：
- 基金本身没有新增直接催化
- 北向或量级证据不够强
- 官方净值尚未出现更连续确认

### 3.3 含费成本与盈亏平衡线（详见 [reference/fee-tiers.md](reference/fee-tiers.md)）

**A 类**（前端申购费，无销售服务费）：
```
含费成本 = 净值成本 + 净值成本 × 0.06%   (天天基金 1 折后)
盈亏平衡卖出净值 = 含费成本 / (1 - 当前档赎回费率)
```

**C 类**（免前端申购费，但有销售服务费）：
```
含费成本 = 净值成本
盈亏平衡卖出净值 = 净值成本 / (1 - 当前档赎回费率)
```

**止损/止盈触发价**：
```
允许亏 X% 的卖出净值 = 含费成本 × (1 - X) / (1 - 赎回费率)
锁定 Y% 利润的卖出净值 = 含费成本 × (1 + Y) / (1 - 赎回费率)
```

**A 类档位常见规则**：
- < 7 天：1.50% 赎回费
- 7-30 天：0.30%
- 30 天-2 年：0.05%
- ≥ 2 年：0.05%（少数免）

**C 类设计意图**：满 7 天免赎回费 + 持续 0.3-0.8% 销售服务费，**适合 7 天 - 1 年的中期持有**。

### 3.4 单基金动作（只能落在以下集合之一）

1. 继续持有
2. 继续持有但不加仓
3. 逢回撤分批加仓
4. 先减仓一档
5. 退出观察仓 / 清仓
6. 暂不动作，等待确认

每个动作都必须配套：
- 触发条件
- 判断理由
- 事实依据
- 信息边界

### 3.5 单持仓报告固定语法

```markdown
## 核心建议

- 当前建议：
- 什么时候动作：
- 判断理由：
- 事实依据：
- 信息边界：
```

这五项缺一不可。

### 3.6 5% 波动率以内的低波固收+置换（详见 [reference/low-wave-bond-funds.md](reference/low-wave-bond-funds.md)）

| 档位 | 5年波动率 | 最大回撤 | 近3年年化 | 候选 |
|------|----------:|--------:|----------:|------|
| A 档（极致低波） | 1.2% | -0.3% | 2.8% | 易方达中短期债券 C、南方中短债 C |
| B 档（平衡型） | 2.5-2.8% | -1.5%~-1.8% | 3.5-4.2% | 招商产业债 C、广发纯债债券 C |
| C 档（攻守兼备） | 3.0% | -2.0% | 3.8% | 富国天利增长债券 C |

决策原则：
- 用户接受度低 → 选 A 档
- 用户想"比余额宝多 1% 收益"（默认）→ 选 B 档
- 用户愿意承受 2% 回撤换更高收益 → 选 C 档

### 3.7 AI 算力建设产业链 5 段归类（详见 [reference/ai-computing-chain.md](reference/ai-computing-chain.md)）

| 段 | 内容 | 代表标的 |
|----|------|---------|
| 上游_芯片本体 | 设备/材料/存储/AI芯片/CPU/GPU/EDA/IP/晶圆代工 | 海光、寒武纪、中微、北方华创、芯原、中芯国际、兆易 |
| 上游_PCB/基材/封装 | PCB、覆铜板、封装设备、精密加工 | 沪电、生益、大族数控、鼎泰高科、生益电子 |
| 中游_整机服务器 | AI服务器整机/白盒/数据中心 | 浪潮信息、紫光股份、中科曙光、工业富联、润泽科技 |
| 中游_光模块/CPO | 光模块/光芯片/光器件/光纤/CPO | 中际旭创、新易盛、源杰、天孚、长飞、永鼎 |
| 中游_网络/IDC | 通信设备/IDC/液冷/电源 | 烽火、亨通、中天、英维克、科华数据 |

**评估补缺时按用户视角的"产业链段"分类**，不用基金报告里给定的"类别"标签。

### 3.8 组合内去重 / 补缺

**重复项识别**：
- 多只宽基 A50 + 沪深 300 + 科创 50 重仓高度重叠（60-80%）→ 砍浮盈最高 + 与其它重叠度最高的
- 红利 + 绿电 + 家电"画像外"叠加 → 优先止盈已修复项

**补缺方向**（按 AI 算力链 5 段扫描）：
- 缺哪段就找该段的 ETF 联接 / 主动基
- 新进基金必须是「场外基金或 ETF 联接」

## 七条铁律（单只基金操作必须遵守）

1. **先读 `投资者行动/持仓情况.md`，再看研究报告** — `基金研究报告/` ≠ 用户真实持仓
2. **占比分母必须含现金**：余额宝 + 银行卡 + 货币基金
3. **用户纯基金路线**：不开证券账户、不做个股；只推荐场外基金或 ETF 联接
4. **基金名引用全称**：6 位代码禁止代指基金本身
5. **不动就是动作**：观察期、等待净值信号也是有效动作
6. **画像 4 档加总必须 = 100%**
7. **列现状 + 目标区间，等用户决策；不要给执行方案** — 不要主动给"先卖 A 再买 B"式的执行顺序

### 第八条铁律

8. **先认可/验证用户已说内容，再继续** — 用户原话：「我还没让你开始算 你就算了我前面说的那些你认可吗？或者你验证了吗？」

### 第九条铁律

9. **基金报告不存在时明说，不要"近似评估"** — 用户原话：「012894 和 010736 没有报告」

### 第十一条铁律

11. **按档合并算，不拆到单只基金** — 用户原话：「为什么把我每一个分类里基金单独算 当然是合一起算」

## 致命错误示例（避免清单）

| 坑 | 症状 | 修正 |
|---|------|------|
| 1 替用户决定档内拆解 | 用户说"按画像"就直接出全表 | 必须问完 4 档再问档内，每步让用户拍 |
| 1.5 用户授权 ≠ 用户拍板 | 用户说"你来算"AI 就开算 | 即使授权也要列待选项让用户拍 |
| 1.6 方向性偏好 ≠ 数值 | "半导体应该最高"AI 直接算百分比 | 让用户拍"哪个段应该占比最高 / 多少 %" |
| 2 画像外独立成档 | 家电 3.9% 单独算 | 任何不在 4 档里的标的必须归到某一档 |
| 3 差值算到每只基金 | 把现金档释放拆到余额宝/银行卡 | 档级差额按 4 档汇总，档内拆解只在用户明确说"展开 X"时再做 |
| 4 推荐报告里没有的产品 | 推荐"光通信主题 ETF 联接"但报告里没有 | 只能用 `基金研究报告/` 目录里已有的 |
| 5 拉数据不指定区间 | 默认"成立至今" | 拉数据前明示"从 X 起拉? 默认近 1 年" |
| 6 跳步给操作建议 | 板块诊断时直接给"加 7,592 元" | 板块诊断只给"缺什么 / 缺多少 / 候选基金" |
| 7 未验证用户前提 | 连续 6 条规则直接套进数字 | 用户 ≥ 3 条规则时必须先回显确认 |
| 8 板块基 vs 主动基 未区分 | 调进攻档时把两者混在一起 | 列当前 9 只基金里哪些是板块基/主动基 |
| 9 限购未事先评估 | 推荐基金后才发现限购 1000/天 | 推荐前先查限购 |
| 10 按标签占比错估产业链暴露 | 用"芯片本体"标签代替"整个半导体产业链" | 按用户视角的产业链段分类 |
| 11 超出方案不给收尾 | 用户选 A 但 A 超画像上限 | 主动附 1-2 个收尾选项 |
| 12 报告"组合总持仓"当总资产 | 占比严重偏离画像 | 占比分母 = 基金市值 + 现金 |
| 13 持仓情况.md 与报告不同步 | 漏算当天加仓 | 以最新报告为准，提示要不要回写 |
