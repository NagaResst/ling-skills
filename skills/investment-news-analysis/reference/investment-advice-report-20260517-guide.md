# 2026-05-17 投资建议报告说明

> 这是 `investment-advice-report-20260517-template.html` 的配套说明，不是独立的“风格参考”。生成任何新的投资建议 HTML 时，都必须直接复用该模板文件作为页面骨架，再按本说明填充内容；禁止脱离模板另写一套 HTML / CSS / 章节结构。

> 📌 **书写规范**：所有报告、预测、验证表格中提及基金时，必须写成“基金名称(代码)”。禁止只写代码，禁止把代码写在名称前面。

## 1. 这版报告的核心价值

2026-05-17 这版不是普通的日报转 HTML，而是把“原始数据 -> 判断 -> 动作”压成了一个适合浏览的决策页面。

它可复用的部分有三层：

1. 页面结构：Hero + 行动看板 + 目录 + 7 个主章节。
2. 决策语法：每只基金统一写成“当前建议 / 什么时候动作 / 判断理由 / 事实依据 / 信息边界”。
3. 视觉样式：米色低刺激背景、卡片式信息块、固定目录、统一表格、基金名称悬浮卡片和风险提示条。

## 2. 页面结构拆解

### 2.1 Hero 区

Hero 区负责三件事：

1. 明确报告身份：这是“持仓分析与建议”的正式归档版本。
2. 告诉用户最少但最关键的上下文：生成日期、覆盖持仓数、归档形式。
3. 在首屏直接给动作分流：今天先复核、接近触发再动作、先不用动。

这一层的目标不是罗列数据，而是先把用户带到“今天该先看哪里”。

### 2.2 目录 + 七章正文

05-17 版本的正文顺序是固定的：

1. 使用边界与画像约束
2. 先看这三件事
3. 为什么这样排优先级
4. 逐只基金怎么处理
5. 复盘与风险提示
6. 下周观察与政策新方向
7. 本次建议摘要

这个顺序有明确用途：

1. 先限定边界，防止建议越权。
2. 再给最短动作列表，降低阅读门槛。
3. 再解释为什么不是别的动作，补上推理链。
4. 然后进入逐只基金卡片，承载细节。
5. 最后补复盘、风险、后续观察和摘要表，方便存档与回看。

补充硬约束：

1. 第 2-6 章必须服务“今天新增了什么”，不能只是把上一日页面换日期。
2. 如果数据基准日没变，必须明确写出“沿用的数据截止日”和“今天新增的新闻 / 风险 / 反向证据”。
3. 第二章至少点出当天新增 item_summaries 的主锚点；第五章必须写清今天新增验证样本数，不能直接复用上一日验证结论。

### 2.3 单基金决策卡

这是整页最关键的组件。每只基金必须落在同一套语法上：

1. 当前建议：一句话结论。
2. 什么时候动作：给出明确触发线。
3. 判断理由：解释动作方向。
4. 事实依据：只写数据、新闻、政策、横向比较。
5. 信息边界：明确哪些结论因为缺数或证据弱而不能上强判断。

如果缺少“信息边界”，报告很容易把弱证据写成强结论。

`什么时候动作` 里的止盈 / 减仓线默认按持仓成本锚定；如果成本没变，就写更接近既有止盈线，不要把线跟着最新净值一起上调。若确需改线，必须在卡片里交代是成本变化还是策略重估。

### 2.4 基金名称悬浮卡片

05-17 原版还有一层容易被忽略，但浏览体验很有用的组件：正文里出现基金全名时，鼠标悬停会弹出小卡片。

这层的作用不是装饰，而是减少来回滚动查表：

1. 让正文里的基金名天然带上下文。
2. 用户不用跳到摘要表就能看到净值、总金额、仓位占比。
3. 长文阅读时，基金名第一次出现就能被快速识别。

原版卡片最小字段只有 3 个：

1. 净值
2. 总金额
3. 占比

这是一个很好的平衡点，再多就会从“提示卡”变成“小面板”。

这层不是可有可无的装饰，而是“基金名称不会在长文里失联”的结构化补丁。只要模板支持悬浮卡片，就应视为正式交付的一部分，而不是后补优化项。

## 3. 章节与数据源映射

### 第一章 使用边界与画像约束

来源：投资者画像、持仓情况、黑名单规则。

必须回答：

1. 这页处理什么，不处理什么。
2. 当前持仓是否命中黑名单。
3. 是否需要证券账户或额外交易条件。

### 第二章 先看这三件事

来源：净值阈值、仓位状态、稳定器角色。

必须回答：

1. 谁要先复核。
2. 哪些仓位接近触发线。
3. 哪些仓位先别动。

### 第三章 为什么这样排优先级

来源：市场量级、阶段判断、缺失数据说明。

必须回答：

1. 市场现在是追高、确认支撑，还是弱修复。
2. 为什么不是现在追仓或砍仓。
3. 当前结论最关键的边界是什么。
4. 本轮已抓到的 ETF 走向全貌是什么，哪些板块强、哪些板块平、哪些板块弱。

如果已经补到核心行业 ETF 和持仓映射 ETF，建议在第三章直接放一张“已获取 ETF 走向总览”表，至少包含：ETF(代码)、来源、前一交易日收盘涨跌幅、走向、对应持仓/备注。若两组 ETF 有重复代码，先去重再展示。

### 第四章 逐只基金怎么处理

来源：latest_navs、benchmark_proxies、news_items、market_scale、单基金深度研究结论。

必须做到：

1. 每只基金都能落到一个动作。
2. 每个动作都有触发线。
3. 每个动作都有事实和边界。
4. 基金列表必须直接来自 `投资者行动/持仓情况.md`，不要手工回忆当前持仓。
5. 卡片数量必须与当前持仓数量完全一致。
6. 若存在止盈线，该线必须能回溯到持仓成本和对应浮盈目标，不能只跟着昨天的上涨一起上移。

### 第五章 复盘与风险提示

来源：历史预测验证结果、当前主要风险。

必须回答：

1. 上一轮判断哪里准、哪里不准。
2. 当前最值得盯防的风险是什么。

### 第六章 下周观察与政策新方向

来源：缺失数据、政策追踪、潜在新主题。

必须回答：

1. 下一轮更新最先补什么数据。
2. 哪些政策方向只能观察，不能直接跳到建仓。

### 第七章 本次建议摘要

来源：第四章结果的压缩版。

作用：

1. 一屏回看全部建议。
2. 方便后续逐日验证。

## 4. 05-17 样式语言总结

### 4.1 颜色

主风格不是高对比金融大屏，而是低刺激的纸面感：

1. 背景：浅米色渐变。
2. 卡片：接近白色，但保留暖色边框。
3. 强调色：砖红色。
4. 风险/观察：暖黄色。
5. 稳定/正向：暗绿色。

### 4.2 组件

固定组件只有这几类：

1. `hero`：首屏摘要。
2. `meta-card`：三格元信息。
3. `action-column`：动作分流。
4. `toc`：左侧目录。
5. `panel`：每个章节卡片。
6. `mini-card`：短说明块。
7. `decision-item`：单基金决策卡。
8. `decision-block boundary`：信息边界块。
9. `note`：统一风险提醒条。
10. `fund-ref` / `fund-card`：正文基金名称悬浮卡片。
11. `table-wrap`：横向摘要表和政策表。

### 4.3 为什么这种样式有效

因为它不是为“炫”服务，而是为“快速找到动作、又能回看证据链”服务。

1. Hero 区先分流动作，降低进入成本。
2. 左侧目录让长文仍可跳读。
3. 卡片化组件把不同信息密度分层。
4. 悬浮卡片让正文中的基金名自带最小信息密度。
5. 单基金区块高度统一，便于横向比较。
6. 在第三章集中放 ETF 走向总览表，可以把“板块当天到底谁强谁弱”一次讲清楚，减少用户在正文里来回拼碎片信息。

## 5. 写作模板硬约束

生成新的投资建议 HTML 时，必须遵守以下约束：

1. 不要把“事实依据”和“判断理由”混写。
2. 不要省略“信息边界”。
3. 不要把政策方向观察直接写成新建仓建议。
4. 第七章摘要必须和第四章逐只卡片一致。
5. 如果北向、ETF、官方净值等基础数据有缺口，必须在第三章和第五章重复交代影响。
6. 如果启用悬浮卡片，正文里必须使用“基金全名(代码)”的统一写法，否则自动注入脚本无法稳定匹配。
7. 目录子链接、第四章基金卡片、第七章摘要表，必须共享同一份持仓基金名单，不能分别手写。
8. 若当前持仓有 9 只基金，目录子链接、第四章卡片和第七章摘要表都必须是 9 个；数量不一致时，优先视为交付失败，而不是小瑕疵。

## 6. 必须使用方式

所有新的投资建议 HTML 都必须复用 05-17 模板，流程是：

1. 先从 `analysis_snapshot_YYYY-MM-DD.json` 提取净值、回撤、浮盈、横向对比。
2. 再把最新新闻和政策压到“理由 / 事实依据 / 信息边界”。
3. 生成 `fund_cards_json`，给悬浮卡片脚本使用。
4. 最后把结果填进 HTML 模板，不得改成另一套页面骨架。

配套模板见同目录：`investment-advice-report-20260517-template.html`。

## 7. 悬浮卡片使用方式

模板已经内置悬浮卡片脚本，不需要逐个手写 HTML 包裹标签，也不允许为了悬浮卡片功能去重写另一套自定义实现。

你只需要提供一个 JSON 数组，格式如下：

```json
[
	{
		"full": "嘉实新能源新材料股票A(003984)",
		"name": "嘉实新能源新材料股票A",
		"code": "003984",
		"nav": "3.1259",
		"amount": "1,191.20 元",
		"weight": "1.28%"
	}
]
```

然后把它填入模板底部的 `{{fund_cards_json}}`。

脚本会自动：

1. 扫描 `.hero` 和 `.content` 区域。
2. 把命中的“基金全名(代码)”替换成带悬浮卡片的 `span`。
3. 跳过目录、链接、表格、脚本、样式和摘要表，避免误包裹。

这意味着：

1. 正文很适合使用基金全名。
2. 摘要表仍保持纯表格，不触发悬浮卡片。
3. 如果基金名缩写、漏代码、或写法不统一，脚本不会命中。

## 8. 交付前对账清单

每次生成 HTML 后，至少做一次数量和名称对账：

1. 从 `投资者行动/持仓情况.md` 提取当前持仓清单，作为唯一基准名单。
2. 检查目录中的基金子链接数量是否等于当前持仓数。
3. 检查第四章 decision-item 数量是否等于当前持仓数。
4. 检查第七章摘要表基金行数是否等于当前持仓数。
5. 检查 `fund_cards_json` 数量是否等于当前持仓数。
6. 检查以上四处出现的基金名称是否全部使用同一套"基金名称(代码)"全称。
7. 检查基金卡片是否正确应用在报告中所有基金名称上
8. 检查悬浮卡片可见性：鼠标悬停正文中的基金名称，确认弹出卡片不被父容器裁切。若不可见，检查 `.decision-item`、`.panel` 是否设置了 `overflow: hidden`——必须为 `overflow: visible`，否则悬浮卡片会被截断。

如果任何一步对不上，不要归因于"模型忘了"，而要回到数据映射和模板填充环节修正，因为这说明页面没有建立单一事实来源。

补充：如果最终产出的 HTML 章节结构、样式骨架或悬浮卡片实现明显不是模板文件这一套，即使内容本身正确，也应视为交付失败，必须回到模板文件重生。

## 9. CSS 注意事项（悬浮卡片相关）

模板中以下 CSS 规则直接影响悬浮卡片能否正常弹出，修改时务必保留：

1. `.decision-item` 必须 `overflow: visible`（不能是 `hidden`，否则弹出卡片被裁切）
2. `.panel` 必须 `overflow: visible`（同理，面板层不能裁切子元素的绝对定位弹出层）
3. `.fund-card` 的 `z-index` 必须足够高（≥9999），确保在所有堆叠上下文中处于最顶层
4. `.action-column` 已设 `overflow: visible`，Hero 区基金名悬浮卡片不会被裁切

如果因为设计需要给其他容器加 `overflow: hidden/auto/scroll`，必须确认该容器内不包含 `.fund-ref` 元素，否则悬浮卡片会被截断。
