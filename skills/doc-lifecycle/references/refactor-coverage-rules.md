# Phase 2a 完整规则 — 重构覆盖率

**适用**：用户已经（或声称已经）做了一波部分重构，要识别哪些文档跟上、哪些没跟上。

---

## 触发条件

**必须**有用户原话或修订记录中明确的"重构事实"。

触发词：
- "我之前重构过一批文档，但没全部完成"
- "哪些文档没重构 / 落伍了 / 没跟进 v2.0"
- "哪些文档没有对齐 v2.0"
- "which docs didn't catch the refactor"

**不适用**：
- 用户没有宣布过重构 → 回退到 `references/scene-audit.md` 做通用质量评分
- 单文件编辑 → 直接 `patch`

---

## 三阶段流程

### Stage 1 — 确立重构基线

用户说"我重构过但没完成"。需要从文档中找到**重构是什么、什么时候发生的**的具体证据。

**1a. 修订记录指纹**：
- 每份文档读前 30 行，提取修订记录表
- 找匹配单一审核人 + 单一日期模式的条目（如全部"AI 校对"于"2026-06-02"，且修订内容形状相似）
- **匹配模式就是基线**

记录：
- 基线日期 D
- 基线审核人 J
- post-baseline 修订内容关键词（这是重构承诺的变更）

**1b. 词汇签名**：
- 如果 post-baseline 条目说"对齐 v2.0: ① 拆 PlanConfig ② 删 crash_recovery ③ `auto_checkout` → `auto_closed`"，**这就是词汇签名**
- pre-baseline 文档预期缺这些字符串
- post-baseline 文档缺这些字符串就是**嫌疑对象**

**Stage 1 输出表**：

| 文档 | 最后审核人 | 最后日期 | 在重构波中? |
|------|----------|---------|----------|
| PRD v1.8 | 用户A | 2026-05-27 | ❌ Pre-baseline |
| 技术方案 v2.0 | AI 校对 | 2026-06-02 | ✅ Post-baseline |
| 安全设计 v1.0 | 用户A | 2026-05-23 | ❌ Pre-baseline |

> ⚠️ Pre-baseline 文档但有 v2.0+ 版本号 → 高度可疑：版本号跳了但审核人没换 = **版本漂移无内容漂移**，需明确标记。

### Stage 2 — 内容采样验证

**仅看日期不够**。

Pre-baseline 日期是必要但不充分：
- 重构前写的文档可能因为 hotfix 偶然含了 post-baseline 词汇
- post-baseline 文档可能声称重构但缺词汇

对每个 pre-baseline 文档，跑快速内容采样：

```bash
for vocab in "${POST_BASELINE_VOCAB[@]}"; do
  echo "=== $vocab in $doc ==="
  grep -c "$vocab" "$doc" || true
done
```

**输出标志**：
- `caught-vocab`：refactor 实际上落地了（即便日期是 pre-baseline，可能是 hotfix）
- `vocab-clean`：真正 pre-baseline 内容
- `mixed`：部分章节重构了，其他没有

**不要停在日期** — 内容验证。用户信任具体证据，不信任元数据。

### Stage 3 — 排序（不是平的清单）

**输出是 triage 清单，不是平铺表**。末尾给推荐的下一步：

"重构优先级（前 2）：[X]（覆盖 [词汇 A, B, C]）和 [Y]（覆盖 [词汇 D]）"

排序依据：
- 文档内容是否影响用户行为（PRD / UI / API → 高）
- 文档是否影响其他文档正确性（DB 设计 / 技术规范 → 高）
- 文档是否叶子（模板 / 变更日志 → 低）

---

## 输出模板

```markdown
## 重构基线
- 日期：D
- 审核人：J
- 承诺变更：① ... ② ... ③ ...

## 文档分级

| # | 文档 | 最后审核人 | 最后日期 | 在重构波中? | 缺失关键术语 |
|---|------|-----------|---------|------------|------------|
| 1 | PRD v1.8 | 用户A | 2026-05-27 | ❌ | auto_closed, @kit.ArkData, NavPathStack |
| 2 | 技术方案 v2.0 | AI 校对 | 2026-06-02 | ✅ | — |
| 3 | 安全设计 v1.0 | 用户A | 2026-05-23 | ❌ | 全套 7 项术语 |

## 重构优先级（从高到低）
1. [doc X] — 影响 [...], 缺失 [...]
2. [doc Y] — 影响 [...], 缺失 [...]
```

---

## Pitfalls

### 不要只信任版本号

"v2.0"在文件名里 ≠ 已重构。要查审核人 + 日期。v2.0 由 `用户A` 在 2026-05-27 写的 = pre-baseline 文档跳了版本号，没跟上重构。

### 不要只信任日期

有些 pre-baseline 文档是短参考表（如"字段速查"），从 post-refactor 主文档派生，**不需要重构**。Stage 1b 内容采样区分"真的没更新"和"始终派生"。

### 不要忘了用户可能也要行动

输出是 triage 清单，**结尾要给推荐的下一步**："前 2 优先重构：[X] 和 [Y]"。不要只扔一张表。

### 不要只列文件名

每份 pre-baseline 文档要配对具体的词汇缺口。"Doc X 是 pre-baseline" 是半个答案。"Doc X 是 pre-baseline，缺 `auto_closed`、`PlanConfig`、`@kit.ArkData`" 才是 actionable。

### 不要把所有 pre-baseline 文档等量齐观

- 用户行为影响文档 → 高
- 影响其他文档正确性 → 高
- 叶子文档 → 低

---

## 完成标准

- [ ] 重构基线（日期 + 审核人）已识别
- [ ] 每份文档已分类（post-baseline / pre-baseline caught-vocab / pre-baseline vocab-clean / pre-baseline mixed）
- [ ] 每份 pre-baseline 文档配对具体词汇缺口
- [ ] "版本漂移无内容漂移" 案例已明确标记
- [ ] 输出末尾有重构优先级列表（不是平表）
