# Phase 3 完整规则 — 5 维评分细则

**适用**：对一批文档独立打分，产出 per-doc scorecard。

---

## 5 维评分框架（详细）

| 维度 | 评估问题 | 评分标准 |
|------|---------|---------|
| A. 结构合规性 | 必备字段/章节是否齐全？章节编号是否连贯？ | 1-5 |
| B. 术语一致性 | 关键术语在**本文档内**是否前后一致？ | 1-5 |
| C. 完整性 | 是否有空章节、TBD、"待补充"？主题覆盖是否充足？ | 1-5 |
| D. 版本号与修订记录 | 文件名版本号 vs 头部版本号？修订记录是否完整？ | 1-5 |
| E. 代码/可落地性 | 含代码的文档，API 是否现代？代码与文档描述是否一致？ | 1-5 |

---

## A. 结构合规性（1-5）

| 分数 | 标准 |
|------|------|
| 5 | 必备字段/章节齐全；章节编号连贯；层级无跳号 |
| 4 | 基本齐全，仅 1-2 处轻微缺漏 |
| 3 | 缺 1-2 个主要章节，或有 1 处章节跳号 |
| 2 | 缺多个主要章节，或有 2+ 处跳号 |
| 1 | 结构严重缺失 |

**对场景 1（新建项目文档）的特殊检查**：
- 8 字段头部是否齐全
- 5 层章节结构是否完整（PRD/SPEC/TECH）

**对场景 2（研读笔记）的特殊检查**：
- 章节有清晰层次
- 末尾有"我还没知道的（待研究）"标注
- README 索引存在

---

## B. 术语一致性（1-5）

**注意：仅检查本文档内的术语一致性，不检查文档间**（那是 Phase 2 的事）。

| 分数 | 标准 |
|------|------|
| 5 | 关键术语（状态枚举、表名、API 名）在本文档内前后完全一致 |
| 4 | 偶尔 1 处术语漂移（如先用旧术语再改新术语） |
| 3 | 2-3 处术语漂移，或同一概念用了 2 个术语 |
| 2 | 多处术语漂移，或术语表与正文不一致 |
| 1 | 术语混乱，难以理解 |

**示例 — 5 分**：`auto_closed` 全篇一致（L23, L45, L67, L89）

**示例 — 2 分**：正文用 `auto_closed`，但术语表条目仍是 `auto_checkout`，第 4 章还出现 `auto_checked_out` 第三个变体

---

## C. 完整性（1-5）

| 分数 | 标准 |
|------|------|
| 5 | 主题覆盖充足；所有章节有实质内容 |
| 4 | 1 处章节内容偏薄但仍完整 |
| 3 | 1-2 处 TBD / "待补充" / 空章节 |
| 2 | 多处空章节或大段 TBD |
| 1 | 文档基本是骨架，无实质内容 |

**示例**：
- 5 分：所有章节都有具体描述、示例、数据
- 3 分：§5.2 标注"待补充"（L412），§6 实际为空（L420-422）
- 1 分：8 个章节，6 个是空骨架

---

## D. 版本号与修订记录（1-5）

| 分数 | 标准 |
|------|------|
| 5 | 文件名版本号 = 头部版本号；修订记录完整对应实际改动 |
| 4 | 版本号一致；修订记录略不完整 |
| 3 | 有一处版本号不一致，或修订记录缺一个版本 |
| 2 | 版本号不一致 + 修订记录不全 |
| 1 | 完全没有版本号或修订记录 |

**示例**：
- 5 分：文件名 `doc_v2.0.md`，头部 v2.0，修订记录含 v1.0 → v1.1 → v2.0 全条目
- 3 分：文件名 v2.0 头部 v2.0 一致，但修订记录缺 v2.0 条目
- 2 分：文件名 v2.0 头部 v1.9 不一致

---

## E. 代码/可落地性（1-5）

**仅当文档含代码时评分**。

| 分数 | 标准 |
|------|------|
| 5 | API 现代（非废弃）；代码语法正确；与文档描述一致 |
| 4 | 代码可用，仅 1 处小瑕疵 |
| 3 | 1-2 处代码用废弃 API，或与文档描述轻微不符 |
| 2 | 多处代码问题或代码与文档严重不一致 |
| 1 | 代码完全无法运行 |

**示例**：
- 5 分：ArkTS 代码语法正确（L89-120）；API 用 `@kit.ArkData` 而非废弃的 `@ohos.router`
- 3 分：用了 `@ohos.router` 一处（L102），其他代码正确
- 1 分：代码示例全是占位符 `<your_code_here>`

---

## 总分计算

**5 维加总**：
- 原始分：1-25
- 标准化：`(总分 / 25) × 10` → 1-10 分制

**决策阈值**：

| 总分（10 分制） | 推荐 |
|--------------|------|
| ≥ 8.5 | A：无需修改 |
| 6 - 8.5 | B：小修小补（针对低分维度） |
| < 6 | C：触发 Phase 4 重写 |

## Pitfalls

### 跨文档引用作评分理由 = 严禁

```markdown
❌ A. 结构 - 3/5 - 比 PRD v2.0 差，缺 8 字段头部
✅ A. 结构 - 3/5 - 缺 8 字段头部中的"审核人"和"版本号"（正文 L1-3）
```

### 临时新增评分维度

如果某份文档有独特问题（如"无章节编号"），**不能**临时新增 F 维度为该文档打分。要么把"F. 章节编号"加到所有文档的评分表（统一性），要么不打分。

### 跳读

11 份 800 行的文档 ≈ 90 分钟纯阅读 — 不能省。

### 聚合分替代单维度分

用户要 per-doc scorecard，**每份文档必须有 5 维细分**，聚合分只在末尾总览表呈现。
