# 场景 3 规范 — 现有文档升级重写

**适用**：把一份或多份 v1.x 文档升级到 v2.x（或 v1.0 → v1.1）。

**产物形态**：仅描述目标版本状态，禁旧留新。

---

## 核心原则

> **新文档只描述新版本的当前状态。**

绝不保留任何形式的"v1.0 → v2.0 对照"、"X → Y 全量替换"、"校对变更"、"v1.0 仅..."、"v1.0 不再..."。

**为什么**：用户要的是"可以作为依据的文档"，不是"迁移报告"。审计轨迹归 git commit，不归文档正文。

---

## 1. 修订记录（极简版）

**只写一行**，不写变更清单：

```markdown
| 版本号 | 修订日期 | 修订人 | 修订内容 | 审核人 |
|--------|----------|--------|----------|--------|
| v2.0 | YYYY-MM-DD | <姓名> | 全文重写对齐 [ref docs] | <姓名> |
```

**❌ 错误写法**：
```markdown
| v2.0 | 2026-06-13 | AI 校对 | ① 表名 X→Y ② 删 crash_recovery ③ auto_checkout→auto_closed ... ⑨ ... | <姓名> |
```

**审计记录的归属**：
- 主记录：git commit message
- 次记录：项目 `CHANGELOG.md`
- 决策日志（如果有）：brainstorm log
- **不在文档正文里**

---

## 2. 术语处理（禁旧留新）

**❌ 错误写法**（保留旧术语作对比）：
```markdown
NavPathStack ... 替代 @ohos.router
v1.0 仅计时型
v1.0 不再涉及 crash_recovery 表
```

**✅ 正确写法**（直接定义新术语）：
```markdown
NavPathStack — 用于页面路由的栈结构。
crash_recovery 表 — 已废弃。
```

**❌ 多列对照表**：
```markdown
| 实体 | v1.0 表述 | v2.0 校对后 |
```

**✅ 单列描述**：
```markdown
| 实体 | v2.0 状态 |
| --- | --- |
```

---

## 3. 实施计划中的"X→Y"行

**❌ 错误**：
```markdown
- [ ] 表名 timing_record → timer_record 全量替换
```

**✅ 正确**：
```markdown
- [ ] DAO / Repository 层使用 timer_record 表
```

---

## 4. 文件名同步更新

**版本号变化必须同步文件名**：

```bash
# v1.0 → v2.0
mv old_doc_v1.0_20260523.md new_doc_v2.0_20260613.md
```

**归档策略（可选）**：
- 如果用户要求保留旧版本作对照 → 旧文件移到 `.archive/v1.x/` 目录
- 否则 → 直接覆盖

---

## 5. 残留扫描（11 类必须=0）

跑 `scripts/check_doc_v2_residue.py`，残留=0 才算完成。

具体 11 类残留模式 → `references/version-rewrite-patterns.md`

---

## 6. 完成标准

- [ ] 新文档只描述目标版本状态
- [ ] 残留扫描脚本退出码 = 0
- [ ] 文件名版本号与头部版本号一致
- [ ] 修订记录 ≤ 1 行
- [ ] 没有 "X→Y"、"v1.0 仅..."、"v1.0 不再..."、"校对变更" 任意模式
- [ ] **用户已确认"这一版就是我想要的"**

---

## 7. Pitfalls

### 7.1 不要给子 agent 传"v1.0 对照表"作为上下文

子 agent 拿到 v1.0 对照表后，默认会"保留 diff 以便审计"。**上下文里只描述目标版本的现状**：

```
新文档正文必须只描述 v2.0 现状。禁止:
- v1.0 表述 / v2.0 校对后的多列对照表
- "X → Y" / "X→Y 全量替换/迁移/修正" 的实施计划行
- "校对变更" / "校对关键变更" blockquote
- "v1.0 仅..." / "v1.0 不再..." 反向引用
- "替代 X" 的术语表条目
- 修订记录里 ⑨ 条变更的展开式列表
```

### 7.2 不要凭印象推断新版应该长什么样

必须先读所有上游 v2.x 文档，提取目标版本的术语/表名/API 名。

### 7.3 残留扫描是必经步骤，不是可选

如果没跑扫描脚本就交付，几乎一定漏掉 "v1.0 仅..." 之类的残留。

### 7.4 用户叫停时立即停

如果用户在 Phase 4 过程中说"先停一下"、"这段我不太满意"、"先别写 §3" — **立即停止当前工作，等用户指示**。Phase 4 不是 batch job。
