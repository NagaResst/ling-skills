# Phase 1 方法 C — 场景 3 完整流程

**适用**：把 v1.x 文档升级到 v2.x（或 v1.0 → v1.1）。

---

## 流程（5 步）

### Step 1 — 读上游基线文档

**不要凭印象推断**新版应该长什么样。

读所有上游 v2.x 文档：
- 数据库设计 v2.x
- 技术方案 v2.x
- API 文档 v2.x
- 已有 v2.x 文档的命名约定 / 8 字段头部 / 5 层章节结构

提取 v2.x 的术语 / 表名 / API 名 / 状态枚举 — 这是新文档必须对齐的目标。

### Step 2 — 写新文档（v2.x 视角）

**假装旧版本从未存在**。

写章节、段落、表格时，把项目当作"从 v2.0 开始"。需要细节时，**交叉引用其他 v2.x 文档**（如"完整数据模型见《数据库设计 v2.0》附录 5.3"），不要在新文档里重复。

### Step 3 — 同步文件名

版本号变化必须同步文件名：

```bash
mv docs/old_doc_v1.0_20260523.md docs/old_doc_v2.0_20260613.md
```

**归档策略**：
- 如果用户要求保留旧版本作对照 → 旧文件移到 `.archive/v1.x/` 目录
- 否则 → 直接覆盖

### Step 4 — 残留扫描（必经步骤）

```bash
python3 ~/.hermes/skills/productivity/doc-lifecycle/scripts/check_doc_v2_residue.py docs/new_doc_v2.0_20260613.md
```

退出码 = 0 表示 11 类残留模式全清零。**退出码 ≠ 0 时，逐条 patch 修复**。

具体 11 类残留模式 → `references/version-rewrite-patterns.md`

### Step 5 — 用户确认

**Phase 4 完成后必须用户确认**："这一版就是我想要的"。

如果用户：
- "OK，提交归档" → 完成
- "§3 要换写法" → 仅 §3 重写，再跑残留扫描
- "整体差不多了，但 §6 要加一节" → §6 重写，再跑残留扫描

**用户叫停时立即停**。Phase 4 不是 batch job。

---

## 完成标准

- [ ] 残留扫描脚本退出码 = 0
- [ ] 文件名版本号与头部版本号一致
- [ ] 修订记录 ≤ 1 行
- [ ] 没有"v1.0 仅..."/"v1.0 不再..."/"校对变更"/"X→Y 全量替换"任意模式
- [ ] **用户已确认"这一版就是我想要的"**

---

## 完成后 → 进入 Phase 2

按 SKILL.md 的路由表决定跑哪些步骤：
- 场景 3 → 跑 Stage 2a（基线判定决定哪些文件要重写）
- 场景 3 → 跑 Stage 2b（升级后与未升级文档做差异检查）
