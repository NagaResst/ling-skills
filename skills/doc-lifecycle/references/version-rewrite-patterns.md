# Phase 4 完整规则 — 11 类残留模式

**适用**：v1.x → v2.x 重写后，扫描残留模式确保新版只描述目标版本状态。

---

## 11 类残留模式（必须全部清零）

| # | 模式 | 检测正则 | 修复方向 |
|---|------|---------|---------|
| 1 | 多列对照表（旧 vs 新） | `\| 实体 \| v1\.0 表述 \| v2\.0 校对后 \|` 等 | 改为单列"v2.0 状态" |
| 2 | "X → Y" 任务行 | `X → Y 全量替换\|X→Y 迁移\|X→Y 修正` | 改写为 v2.0 任务 |
| 3 | "校对变更" blockquote | `> 校对变更：` / `> 校对关键变更：` | 删除；如非说不可，改写为 v2.0 现状陈述 |
| 4 | "v1.0 仅..." 反向引用 | `v1\.0\s*仅` | 改写为 v2.0 现状 |
| 5 | "v1.0 不再..." 反向引用 | `v1\.0\s*不再` | 改写为 v2.0 现状 |
| 6 | "v1.0 → v2.0 关键差异汇总" 章节 | 标题含"v1.0 → v2.0 关键差异" | 删除；如必要改 git 历史 |
| 7 | "替代 X" 术语表条目 | `替代 X\|X 的替代` | 直接定义术语，不写"替代" |
| 8 | 修订记录 ⑨ 条变更列表 | 修订内容列含"①...②...③..."多于 5 条 | 一行极简 |
| 9 | "v1.0 表述 / v2.0 校对后" 章节 | 章节标题含此模式 | 删除 |
| 10 | "X 由 v1.0 改为 v2.0" 叙述 | `由 v1\.0 改为 v2\.0\|从 v1\.0 改为 v2\.0` | 改写为 v2.0 现状 |
| 11 | "v1.0 不再涉及..." 段落 | `v1\.0\s*不再\s*涉及\|v1\.0\s*已经\s*删除` | 改写为 v2.0 现状 |

## 通用残留检测脚本

`scripts/check_doc_v2_residue.py` 已实现上述 11 类模式的自动化扫描。

**用法**：

```bash
python3 ~/.hermes/skills/productivity/doc-lifecycle/scripts/check_doc_v2_residue.py docs/new_doc_v2.0.md
```

**退出码**：
- 0 = 全部清零
- 1 = 还有残留（脚本会打印模式名 + 行号）
- 2 = 参数错误

## 用户特定旧术语注入

如果项目有特定旧术语（如 `timing_record`、`crash_recovery`、`auto_checkout`），脚本支持通过 `--old-terms` 参数注入：

```bash
python3 check_doc_v2_residue.py --old-terms timing_record,crash_recovery,auto_checkout docs/new_doc_v2.0.md
```

注入后，模式 12-99（用户自定义）也会被扫描。

## 11 类模式的修复模式

### 模式 1: 多列对照表

**❌ 错误**：
```markdown
| 实体 | v1.0 表述 | v2.0 校对后 |
| --- | --- | --- |
| Project | 含 PlanConfig 字段 | 拆出独立表 |
```

**✅ 正确**：
```markdown
Project 表 — 包含项目基础字段（详见《数据库设计 v2.0》§3.1）。
PlanConfig — 独立表（详见《数据库设计 v2.0》§3.5）。
```

### 模式 2: "X → Y" 任务行

**❌ 错误**：
```markdown
- [ ] 表名 timing_record → timer_record 全量替换
```

**✅ 正确**：
```markdown
- [ ] DAO / Repository 层使用 timer_record 表
```

### 模式 3: "校对变更" blockquote

**❌ 错误**：
```markdown
> 校对变更：v1.0 写为 X，v2.0 统一改为 Y。
```

**✅ 正确**：
```markdown
Y — 用于 [用途]。
```

### 模式 4/5: "v1.0 仅..." / "v1.0 不再..."

**❌ 错误**：
```markdown
v1.0 仅计时型
v1.0 不再涉及 crash_recovery 表
```

**✅ 正确**：
```markdown
计时型 — 用于 [用途]。
crash_recovery 表 — 已废弃。
```

### 模式 6: "v1.0 → v2.0 关键差异汇总" 章节

**❌ 错误**：
```markdown
## 附录：v1.0 → v2.0 关键差异汇总
[详细对照表]
```

**✅ 正确**：
删除整个章节。如必要，回到 git history 查 v1.0 → v2.0 变更。

### 模式 7: "替代 X" 术语表条目

**❌ 错误**：
```markdown
NavPathStack — 用于路由的栈结构，替代 @ohos.router。
```

**✅ 正确**：
```markdown
NavPathStack — 用于路由的栈结构。
```

### 模式 8: 修订记录 ⑨ 条变更列表

**❌ 错误**：
```markdown
| v2.0 | 2026-06-13 | AI 校对 | ①表名 X→Y; ②删 crash_recovery; ③auto_checkout→auto_closed; ⑨... | <审核人> |
```

**✅ 正确**：
```markdown
| v2.0 | 2026-06-13 | AI 校对 | 全文重写对齐 [ref docs] | <审核人> |
```

### 模式 9-11: 类似，按上述"删除旧引用 + 重写为 v2.0 现状"原则处理

## 不适用此规则的情况

这些是例外，**不**清零：

- **迁移指南**：文档存在就是为了记录 v1 → v2 过渡，应保留对照表
- **CHANGELOG.md**：按定义是变更日志
- **PR description**：差异式摘要本就预期
- **审计报告**：用户明确要求审计

**本规则适用于**：权威参考文档（设计文档、需求文档、技术方案、API 文档、代码规范、安全设计等）— 未来开发者读这些作为"X 是如何工作的"事实之源。

## 完成标准

- [ ] 11 类残留扫描退出码 = 0
- [ ] 用户特定旧术语扫描退出码 = 0（如适用）
- [ ] 文件名版本号与头部版本号一致
- [ ] 修订记录 ≤ 1 行
- [ ] **用户已确认"这一版就是我想要的"**

## Pitfalls

### 不要给子 agent 传"v1.0 对照表"作为上下文

子 agent 默认会保留 diff 作审计。上下文里**只描述目标版本的现状**。

### 不要凭印象推断新版应该长什么样

必须先读所有上游 v2.x 文档。

### 残留扫描是必经步骤，不是可选

不跑脚本 = 几乎一定漏掉 "v1.0 仅..." 之类的残留。

### 用户叫停时立即停

Phase 4 不是 batch job。
