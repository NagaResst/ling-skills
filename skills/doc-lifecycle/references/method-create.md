# Phase 1 方法 A — 场景 1 完整流程

**适用**：从零为新项目创建文档体系。

---

## 流程（7 步）

### Step 1 — 评估范围

确定项目类型：手机 App / Web 服务 / 库 / 工具脚本 / ...

不同的项目类型有不同的命名约定和文档类型清单。**先确定再创建**。

### Step 2 — 创建目录结构

标准结构（按项目生命周期排列）：

```
docs/
├── 00_规范文档/          # 底层支撑（规范文档）
├── 01_需求文档/          # 上游决策
├── 02_设计文档/          # 中游设计
├── 03_技术文档/          # 中游实现
├── 04_测试文档/          # 下游验证
└── 05_项目管理/          # 进度/会议
```

**编号规则**：数字越小越接近上游决策。00 是底层支撑，01-05 按生命周期排列。

### Step 3 — 确定命名规范

**先确定项目缩写和日期格式，再创建文件**。

- 项目缩写：项目名首字母缩写（如 `MyApp` = `MA`）
- 文件名日期：`YYYYMMDD`（8 位，无分隔符）
- 头部日期：`YYYY-MM-DD`（利于人类阅读）

中途改格式 = 所有文件名重命名 + 所有头部修改。**先确定再创建**。

### Step 4 — 创建文档（带 8 字段头部）

每份文档的第一段就是 8 字段头部（详见 `references/scene-create.md` §2）。

不要先创建空文件再回头补 — **写文件时第一段就是头部**。

### Step 5 — 应用 5 层章节结构

PRD / SPEC / TECH 等主要文档推荐 5 层结构。TEST / PM 文档可裁剪（详见 `references/scene-create.md` §4）。

### Step 6 — 用户反馈循环

详见 `references/scene-create.md` §6。重点：

- 用户单字段修正 → 用 `patch` 工具精准改一处
- 用户拒绝某特性 → 视为需求变更信号，立即重写相关章节，不询问"是否确认"
- 用户提供完整文档规范 → 直接采用为权威规范

### Step 7 — 命名/头部自动化检查

```bash
python3 ~/.hermes/skills/productivity/doc-lifecycle/scripts/check_doc_header.py docs/**/*.md
python3 ~/.hermes/skills/productivity/doc-lifecycle/scripts/check_doc_naming.py docs/**/*.md
```

退出码 = 0 表示全部合规。

- [ ] 命名符合 `[项目缩写]_[文档类型]_[模块]_[版本号]_[日期].md`
- [ ] 修订记录表紧跟头部
