# 场景 1 规范 — 新建项目文档体系

**适用**：从零为新项目创建文档体系（PRD/SPEC/TECH/TEST 等）。

**产物形态**：8 字段头部 + 5 层章节结构 + 命名规范。

---

## 1. 命名规范

**文件名**：
```
[项目缩写]_[文档类型]_[模块]_[版本号]_[日期].md
```

- 项目缩写：项目名首字母大写缩写（如 `GYC` = 项目名称）。**先确定缩写再创建文件**，中途改格式会导致所有文件名需要重命名。
- 文档类型：`PRD` / `SPEC` / `TECH` / `TEST` / `API` / `DB` / `DESIGN` 等
- 模块：可选，如 `PRD_产品需求_v1.0.md` / `SPEC_计时_v1.0.md`
- 版本号：`vX.Y` 或 `vX.Y.Z`
- 日期：8 位数字 `YYYYMMDD`（利于文件名排序）

**示例**：`MyApp_PRD_产品需求_v1.0_20260523.md`

---

## 2. 8 字段头部（必填，不可缺）

每份文档第一段必须是这 8 个字段（顺序固定）：

```markdown
**项目名称**：<项目全称>
**文档类型**：<需求文档 / 设计文档 / 技术方案 / 测试用例 / ...>
**创建日期**：YYYY-MM-DD
**创建人**：<姓名>
**审核人**：<姓名>
**版本号**：vX.Y.Z
**状态**：<草稿 / 审核中 / 已发布 / 已归档>
```

**注意**：
- 正文头部日期格式：`YYYY-MM-DD`（利于人类阅读），与文件名 `YYYYMMDD` 不同
- 创建人/审核人**不能是占位符**（如 `[姓名]` / `张三`），用真实姓名或会话硬编码
- 8 字段后**立即跟修订记录表**，不留空行

---

## 3. 修订记录表（紧跟头部）

```markdown
| 版本号 | 修订日期 | 修订人 | 修订内容 | 审核人 |
|--------|----------|--------|----------|--------|
| v1.0 | YYYY-MM-DD | <姓名> | 初稿创建 | <姓名> |
```

---

## 4. 5 层章节结构（推荐，但可裁剪）

PRD / SPEC / TECH 等主要文档推荐用 5 层结构，TEST / PM 文档可裁剪：

```
1. 项目概述
   1.1 项目背景
   1.2 项目目标
   1.3 适用范围
   1.4 术语定义

2. 需求分析
   2.1 用户画像
   2.2 业务场景
   2.3 功能需求
   2.4 非功能需求

3. 详细设计
   3.1 架构设计
   3.2 模块设计
   3.3 数据设计
   3.4 接口设计

4. 实施计划
   4.1 开发周期
   4.2 里程碑
   4.3 风险评估

5. 附录
   5.1 参考资料
   5.2 相关文档
```

**裁剪原则**：TEST 文档可省 §3 详细设计、PM 文档可省 §2 需求分析。框架通用，但允许按文档性质裁剪。

---

## 5. 版本号升级规则

| 修改性质 | 版本号变化 | 修订记录 |
|---------|-----------|---------|
| 需求/规格变更 | +0.1（如 v1.0 → v1.1） | 是 |
| 新增章节 | +0.1 | 是 |
| 技术澄清（不影响需求/规格） | 不变 | 否 |
| 单纯文字修订 | 不变 | 否 |

**版本号同步**：版本号变化必须**同步更新文件名**（不只是头部）。

---

## 6. 用户反馈循环纪律

- 用户对单字段的修正 → 用 `patch` 工具精准改一处，不要重写整个文档
- 用户修正命名约定（如"不叫 A，叫 B"）→ 立即重命名所有受影响文件 + 更新所有交叉引用
- 用户拒绝某个特性 → 视为 PRD 需求变更信号，**立即重写相关章节**，不需要询问"是否确认"
- 用户提供完整的文档规范 → **直接采用为权威规范**，不要自由发挥

---

## 7. 完成后必做

- 用 `scripts/check_doc_header.py` 校验 8 字段头部齐全
- 用 `scripts/check_doc_naming.py` 校验文件名符合 `[项目缩写]_[文档类型]_[模块]_[版本号]_[日期].md`
- 报告输出：变更摘要表（不要把 diff 当 narrative 念）
